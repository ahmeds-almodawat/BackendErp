# v373 RLS Closure

## Purpose

v373 turns the broad v366 RLS warning into a concrete closure registry. The goal is not to pretend every RLS policy is finished. The goal is to make every RLS-enabled table visible, classified, and tied to a required production test.

## What this patch adds

- `src/engines/enterpriseV373RlsClosureEngine.ts`
- `scripts/qa-v373-rls-closure.mjs`
- `docs/V373_RLS_CLOSURE_REPORT.md` generated by QA
- `docs/V373_RLS_CLOSURE_REGISTRY.json` generated by QA
- `qa:v373` wired into `qa:all`

## Closure statuses

| Status | Meaning |
|---|---|
| `explicit-policy` | Static SQL policy was detected for the table. |
| `backend-rpc-only` | Direct browser access is not trusted; access should be through permission-checked RPC/functions only. |
| `module-policy-test-required` | Table likely needs explicit module-level branch/company/read/write tests. |
| `guardrail-policy-review` | Table needs owner review before production enablement. |
| `unresolved` | Scanner could not classify the table; QA fails if this appears. |

## Production rule

A table is not production-closed merely because RLS is enabled. For production, each table must have one of the following:

1. Explicit SQL policies with role/scope tests.
2. Backend/RPC-only access with permission, branch/store/company, and audit tests.
3. A documented module-level policy test that proves allowed/denied behavior.
